# Словарь

## Предметная область

Дайвер-любитель - пользователь, ведущий личный логбук, ищущий дайв-сайты и планирующий погружения.

Инструктор / дайв-гид - профессиональный пользователь, управляющий студентами, курсами, группами и дайв-турами.

Дайв-центр - организация, публикующая услуги, курсы, туры, слоты бронирования и работающая с клиентскими запросами.

Администратор - пользователь административного контура, выполняющий модерацию, блокировки, аудит и контроль качества данных.

Логбук - цифровой журнал погружений пользователя.

DiveLogEntry - запись логбука с данными о погружении, условиях, профиле глубины, газовой смеси и связанными метаданными.

Дайв-компьютер / Dive computer - физическое устройство дайвера, передающее профиль погружения в мобильное приложение через BLE, USB или SDK производителя.

Дайв-сайт / DiveSite - локация для погружений с координатами, типом, сложностью, описанием, отзывами и статусом модерации.

План погружения / DivePlan - план будущего погружения с входными параметрами, расчетами, предупреждениями и версией алгоритма.

Бездекомпрессионный предел / NDL - расчетный предел времени пребывания на заданной глубине без обязательной декомпрессионной остановки; используется планировщиком как справочный параметр и не заменяет обучение или профессиональную оценку.

Booking - бронирование дайв-тура или курса, проходящее через статусы Requested, AcceptedByDiveCenter, PaymentPending, Confirmed и другие.

PaymentIntent / payment intent - платежное намерение, созданное через внешнего платежного провайдера.

Premium-функции - платные возможности платформы, доступ к которым подтверждается через внешний payment provider без хранения платёжных реквизитов в платформе.

Уведомление / Notification - сообщение пользователю через push, in-app, email или SMS.

Кейс модерации / ModerationCase - кейс, создаваемый по жалобе, предложению дайв-сайта или административному действию.

Статистика использования - агрегированные показатели активности платформы для администратора; относится к аналитике и отчетности, а не к доменным событиям, изменяющим состояние предметной области.

## Архитектурные термины

Bounded Context / bounded context - граница доменной модели, внутри которой термины, агрегаты и правила имеют однозначный смысл.

Aggregate - группа связанных сущностей и value objects с едиными инвариантами и точкой изменения состояния.

Доменное событие / Domain Event - событие, фиксирующее уже произошедший бизнес-факт, например `DiveLogCreated` или `BookingConfirmed`.

Команда / Command - намерение изменить состояние системы, например `ImportDiveComputerData` или `RequestBooking`.

Политика / Policy - правило реакции на событие, инициирующее команду, уведомление, индексацию или другую побочную обработку.

Проекция чтения / Read Model - производная модель чтения, оптимизированная для поиска, ленты или отображения; не является источником истины.

Modular Monolith - стартовый архитектурный стиль backend, при котором домены изолированы логически, но поставляются как единый runtime.

Шина событий / Message Broker - инфраструктурный компонент для асинхронной доставки доменных событий.

Outbox - подход, при котором изменение доменного состояния и запись события фиксируются в одной транзакции перед публикацией.

SyncState / syncStatus - техническое состояние синхронизации offline-данных: LocalOnly, PendingSync, Syncing, Synced, Conflict, SyncFailed.

## Единообразные определения

Search / Geo-indexing - производная read model для геопространственного и полнотекстового поиска; не источник истины.

Admin Module - домен, координирующий moderation cases и audit log; не владеет первичным контентом Social, Dive Sites или Booking.

Notifications - домен доставки уведомлений; события доставки имеют продуктовый смысл, но не откатывают исходные доменные транзакции.

PaymentFailed - доменное событие платежа, а не основной статус `Booking`.

FeedProjection - read model социальной ленты, а не агрегат.
